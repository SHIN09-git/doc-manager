# P2 第三轮商业化施工文档

本文件承接 `P2_ROUND2_COMMERCIALIZATION_BUILD_PLAN.md`。前两轮已经补齐邮件投递、签名支付 webhook、管理汇总 API、管理后台二级页面、支付事件适配、组织导出和备份校验。第三轮目标是把灰度运营能力继续推进到可试收费前的“外部服务实接与运维闭环”。

## 施工顺序

```text
1. 邮件服务商实接说明与模板配置
2. 支付渠道适配器落地
3. 备份失败告警与恢复演练文档
4. 管理后台批量操作与反馈流转增强
5. PostgreSQL Store 生产级拆分方案
6. 文档与测试更新
7. 施工后 review
```

## P2R3-1：邮件服务商实接说明与模板配置

要做：
- 增加至少一个真实邮件服务商的 webhook 接入示例。
- 将验证邮件、重置邮件模板配置项文档化。
- 记录退信、限流、投递失败的处理建议。

验收：
- 部署文档中有可复制的邮件服务商配置示例。
- 邮件 webhook 请求体字段清晰。
- 生产环境邮件失败不会暴露内部错误给用户。

## P2R3-2：支付渠道适配器落地

要做：
- 在现有 payment adapter 之上拆出 provider 映射函数。
- 至少给一个渠道写适配示例。
- 明确套餐 plan 与渠道价格 ID 的映射。
- 补充支付失败、退款、取消订阅的运营提示。

验收：
- 支付事件从渠道格式归一为内部事件。
- 价格 ID 不能随意写入 plan。
- 退款和支付失败不会误降级。

## P2R3-3：备份失败告警与恢复演练文档

要做：
- 为备份脚本增加失败时的退出码和明确错误输出。
- 增加备份恢复演练文档。
- 增加建议的计划任务配置示例。

验收：
- 备份和校验失败时 CI/计划任务可以感知。
- 文档说明恢复前需要人工确认和只读校验。

## P2R3-4：管理后台增强

要做：
- 反馈列表增加更清晰的状态筛选。
- 最近错误增加复制详情入口。
- 用量和审计支持导出 CSV。
- 成员高风险操作增加二次确认文案。

验收：
- 管理后台能完成灰度运营的日常查看和导出。
- 高风险操作有确认。

## P2R3-5：PostgreSQL Store 生产级拆分方案

要做：
- 出具表级 repository 改造设计。
- 优先拆出高频表：`documents`、`writer_profiles`、`ai_usage`、`audit_logs`。
- 设计迁移版本表和增量迁移脚本结构。

验收：
- 有明确的拆分顺序和兼容策略。
- 不在本轮大改全部 Store，但为 P3 实施准备好方案。

## 非目标

本轮不做：
- 完整支付结算页面和发票系统。
- 企业级 SLA 自动化。
- 多租户高并发表级 repository 全量实现。
- 第三方工单系统深度集成。
