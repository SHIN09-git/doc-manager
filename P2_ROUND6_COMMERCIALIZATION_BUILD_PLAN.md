# P2 第六轮商业化施工计划

更新时间：2026-05-24

## 2026-05-26 施工完成记录

本轮已按计划完成阶段 A-E 的 MVP 落地，并补充阶段 F 的主要验证：

- 阶段 A：`/api/ops/recent-errors` 已收口为仅返回当前组织的系统错误和 AI 失败记录；平台级 `organization_id === null` 事件不再暴露给普通组织后台；跟进接口也拒绝处理全局事件。
- 阶段 B：新增 `ops_triage` 轻量跟进映射，AI 失败记录和系统错误均可通过同一跟进入口保存状态、负责人、备注、优先级和 SLA；新增 `POST /api/feedback/batch-status`，反馈批量处理不再由前端逐条请求完成。
- 阶段 C：新增 `AI_COST_RATES`、`AI_DAILY_BUDGET_CNY`、`AI_MONTHLY_BUDGET_CNY` 配置；`recordUsage` 写入统一 `estimated_cost`；后台展示今日/本月成本、任务类型成本分布和预算状态。
- 阶段 D：新增 `admin_preferences`，后台审计筛选、错误筛选和反馈筛选可按组织与用户保存；前端保留 localStorage 兜底，并提供筛选导出/清空入口。
- 阶段 E：后台概览新增 SLA 超时/临近摘要和一键复制运营日报；反馈与错误列表新增 SLA 筛选。
- 阶段 F：补充后端 API 测试与独立后台 E2E 覆盖，已通过 `npm run check`、`npm test`、`npm run test:e2e`、`npm audit --omit=dev`、`git diff --check`。

剩余未纳入本轮的事项：真实支付渠道 SDK/签名算法接入、平台级超级管理员后台、PostgreSQL 写入路径 repository 化。

本计划承接第五轮“后台深水区”：用量趋势、成本估算、审计保存筛选、反馈/错误负责人备注 SLA 已完成。第六轮目标是把独立后台从“可运营查看与轻量跟进”推进到“更安全、更可协作、更可计费分析”的灰度运营后台。

## 一、背景与 Review 结论

本轮复查的重点文件：

- `src/admin/adminPage.js`
- `server/src/app.js`
- `styles.css`
- `e2e/workbench.spec.js`
- `server/tests/commercial-api.test.js`

主流程结论：

- 独立后台用量、审计、反馈、错误四个深水区入口可用。
- 反馈和错误跟进字段已落到后端元数据，不只是前端临时状态。
- E2E 已覆盖用量趋势、成本估算、审计保存筛选、反馈跟进、错误跟进。

需要下一阶段优先处理的风险：

- `/api/ops/recent-errors` 当前会返回 `organization_id === null` 的全局系统事件；组织管理员不应默认看到平台级全局事件。独立后台当前使用 `/api/admin/dashboard`，主路径没有暴露该列表，但接口边界需要收口。
- 错误跟进当前只支持 `system_events`，而 `/api/ops/recent-errors` 中的 `ai_usage` 失败记录虽然带了 `id`，但不能通过同一个 triage 接口更新，接口契约需要统一。
- 审计保存筛选当前只保存在浏览器本地，未按组织/用户同步，团队运营场景不可共享。
- 反馈批量更新仍是前端逐条请求，数据量上来后不够稳。
- 成本估算仍使用前端默认单价兜底，正式商业化需要后端模型单价表、月度账单口径和预算预警。

## 二、施工原则

- 先收权限边界，再扩功能。
- 不引入完整工单系统，先保持轻量事件跟进模型。
- 所有运营动作必须有审计记录。
- 本地模式不受影响，云端后台能力只在登录后启用。
- 新增字段保持向后兼容，旧 JSON Store 数据不能崩。

## 三、阶段 A：运营事件权限边界收口

目标：避免组织管理员看到或修改平台级全局事件。

任务：

- 调整 `/api/ops/recent-errors`：默认只返回当前组织事件和当前组织 AI 失败记录。
- 对 `organization_id === null` 的平台级事件增加明确策略：暂不对组织管理员展示；后续若需要平台运营后台，再单独设计超级管理员权限。
- 调整 `POST /api/ops/events/:eventId/triage`：只允许处理当前组织归属事件。
- 补充跨组织测试：A 组织管理员不能查看或跟进 B 组织事件，也不能处理全局事件。
- 更新 `SECURITY.md` 的运营后台权限边界说明。

验收：

- 普通成员仍然 403。
- owner/admin 只能看到本组织错误事件。
- 全局事件不会通过组织后台泄漏邮箱、URL、错误详情等敏感元数据。

## 四、阶段 B：错误与反馈轻量工单接口统一

目标：让反馈、系统错误、AI 失败都能用一致的跟进字段处理。

任务：

- 新增后端批量反馈状态接口，例如 `POST /api/feedback/batch-status`。
- 为 AI 失败记录设计轻量跟进存储策略：优先在 `ai_usage` 增加 `metadata` 或新增 `ops_triage` 映射表，避免把系统事件和用量记录强行混用。
- 统一前端跟进字段：状态、优先级、负责人、备注、SLA。
- 所有跟进保存写审计日志。
- 增加批量跟进测试和错误路径测试。

验收：

- 反馈批量处理不再由前端逐条请求完成。
- `recent-errors` 中展示的每类错误都有明确可保存路径，不能出现“能显示但保存失败”的记录。
- 旧数据缺少跟进字段时仍显示“未分配 / 未设置”。

## 五、阶段 C：成本估算后端化与预算预警

目标：让成本展示从前端估算变成后端统一口径。

任务：

- 新增模型价格配置，例如 `AI_COST_RATES`，按 provider/model 配置输入/输出 token 单价。
- `recordUsage` 写入时计算 `estimated_cost`，失败请求保留 0 或只计 prompt 成本的明确策略。
- 后台用量页增加日/月成本汇总、按任务类型成本分布。
- 新增组织预算阈值配置草案：日预算、月预算、接近阈值告警。
- 文档说明估算成本不是发票金额，正式账单以支付渠道和套餐为准。

验收：

- 用量历史接口返回的 `estimated_cost` 有后端统一来源。
- 前端默认单价只作为旧数据兜底。
- 测试覆盖不同模型价格和缺省价格。

## 六、阶段 D：后台偏好与审计筛选云端化

目标：让运营筛选可以随账号和组织保存。

任务：

- 新增管理员偏好接口，保存审计筛选、错误筛选、常用日期范围。
- 筛选记录按 `organization_id + user_id` 隔离。
- 独立后台从 localStorage 迁移为优先读云端，离线或失败时退回本地。
- 增加导出/清空个人后台偏好入口。

验收：

- 同一管理员换浏览器后可继续使用保存筛选。
- 不同组织之间不会串用筛选条件。
- API 失败时后台仍可使用本地筛选兜底。

## 七、阶段 E：后台可观测与 SLA 工作流

目标：让后台能主动暴露需要处理的运营风险。

任务：

- 错误和反馈列表增加“SLA 即将到期 / 已逾期”筛选。
- 概览页增加待处理反馈、处理中错误、逾期 SLA 数量。
- 增加一键复制运营日报：用量、失败、反馈、错误、账单事件。
- 预留告警 webhook：SLA 逾期、AI 失败激增、邮件退信激增。

验收：

- 后台首页能一眼看到“今天要处理什么”。
- SLA 逾期能通过筛选和摘要进入运营视野。

## 八、阶段 F：验证与文档收口

- 补充后端跨组织权限测试。
- 补充独立后台 E2E：保存筛选云端化、批量反馈、AI 失败跟进、SLA 筛选。
- 更新 `README.md`、`SECURITY.md`、`ARCHITECTURE.md`、`TODO.md`、`REVIEW.md`。

## 九、不在本轮处理

- 真实支付渠道 SDK 接入仍按之前安排暂缓。
- 不做完整客服工单系统。
- 不做平台超级管理员后台，除非先明确部署形态和权限边界。
- 不改主工作台布局。
