# P1 商业化施工文档

本文件承接 `P0_COMMERCIALIZATION_BUILD_PLAN.md`。P0 已完成最小商业化底座：账号、组织、云端文档、云端执笔人、后端 AI 代理、用量统计、API Key 加密、审计和前端云端入口。

P1 的目标不是继续堆功能，而是把 P0 从“可验证原型”推进到“可灰度上线的商业服务”。

## 施工顺序

```text
1. 持久化数据库与迁移
2. 账号安全与组织协作
3. 云端同步冲突处理
4. AI 代理生产化
5. 配额、套餐与付费预埋
6. 部署、监控与备份
7. 管理后台与运营工具
8. 合规与用户协议
9. 上线灰度验收
```

## P1-1：数据库生产化

当前 `server/.data/db.json` 适合 P0 验证，不适合多人商业环境。

要做：
- 迁移到 PostgreSQL 或兼容托管数据库。
- 建立 migration 目录和版本化迁移脚本。
- 为 `organization_id`、`owner_id`、`handle`、`created_at` 建索引。
- 保留 JSON store 作为本地开发模式。
- 增加数据导入脚本，把 P0 JSON 数据迁移到数据库。

验收：
- 云端文档、执笔人、版本、用量、审计都从数据库读写。
- 单测可以使用临时测试库或内存替身。
- 旧 JSON 数据可以迁移一次。

## P1-2：账号安全与组织协作

要做：
- 邮箱验证。
- 忘记密码与重置密码。
- 组织成员邀请。
- 角色权限细化为 owner/admin/member。
- Session 轮换和退出所有设备。
- 登录失败节流。

验收：
- 未验证邮箱不能进入付费组织能力。
- 非组织成员不能访问组织数据。
- member 不能管理 API Key 和成员。

## P1-3：云端同步冲突处理

P0 是手动同步和覆盖保存。P1 需要避免误覆盖。

要做：
- 为文档和执笔人增加 `version` 或 `etag`。
- 前端保存时携带本地最后同步版本。
- 云端版本更新后，前端提示“覆盖云端 / 另存副本 / 拉取云端”。
- 执笔人重训和手工编辑说明文件时保留版本差异。

验收：
- 两个设备同时编辑不会静默覆盖。
- 冲突时用户能看见差异摘要。

## P1-4：AI 代理生产化

要做：
- 支持组织默认模型和任务级模型。
- 支持平台托管 Key 与用户自带 Key 两种模式。
- 增加请求超时、取消、失败重试和供应商错误映射。
- 记录 prompt_tokens、completion_tokens、成本估算。
- 给执笔人构建、文档生成、PPT 生成分别打 `task_type`。

验收：
- 前端所有 AI 调用都可选择走云端代理。
- 后端不会向前端返回明文 Key。
- 用量面板能按任务类型查看消耗。

## P1-5：套餐与付费预埋

先做套餐和限制，不急着接支付。

要做：
- `organizations.plan` 支持 free/pro/team。
- 配额按套餐读取。
- 达到配额时返回可识别错误码。
- 前端显示剩余额度和升级入口占位。
- 预留支付 webhook 表结构。

验收：
- free 组织能被每日额度限制。
- 前端能展示额度不足提示。

## P1-6：部署、监控与备份

要做：
- Dockerfile 或部署脚本。
- 健康检查和 readiness check。
- 结构化日志。
- 错误监控接入点。
- 数据库定时备份。
- 环境变量校验，生产环境禁止默认密钥。

验收：
- 生产环境缺少强密钥时服务拒绝启动。
- 能查看最近 AI 错误、接口错误、登录失败。

## P1-7：管理后台

要做：
- 组织基础信息管理。
- 成员列表和角色调整。
- API Key 配置与禁用。
- 用量报表。
- 审计日志查询。

验收：
- owner 能完成团队最常见管理动作。
- 审计日志可按动作和时间过滤。

## P1-8：合规与协议

要做：
- 隐私政策。
- 用户协议。
- 数据删除说明。
- AI 生成内容免责声明。
- 样本文档和执笔人训练数据的隐私提示。

验收：
- 注册页、设置页、导入样本文档前都有必要提示。
- 用户能导出或删除自己的数据。

## P1-9：灰度上线

要做：
- 内测账号白名单。
- 种子用户反馈入口。
- 日志留存策略。
- 错误恢复流程。
- 回滚脚本。

验收：
- 可以邀请 3-5 个真实用户试用。
- 出现后端故障时，本地模式仍可打开并继续编辑。
