# P1 第二轮商业化施工文档

本文件承接 `P1_COMMERCIALIZATION_BUILD_PLAN.md` 第一轮完成记录。第一轮已经补齐生产环境校验、存储抽象预留、组织邀请、同步版本冲突、AI 任务类型和套餐配额基础能力。

第二轮目标：把“可灰度验证”的云端模式继续推进到“可邀请真实小团队试用”的状态。优先补账号安全、真实数据库落地、部署可观测和最小管理能力。

## 施工顺序

```text
1. 账号安全闭环
2. PostgreSQL 真实存储
3. 团队管理增强
4. 部署与可观测
5. 合规与数据删除
6. 灰度试用准备
```

## P1R2-1：账号安全闭环

要做：
- 增加邮箱验证表和验证接口。
- 增加忘记密码、重置密码和重置令牌过期机制。
- 增加登录失败节流。
- 增加 Session 轮换和退出所有设备。
- 前端云端面板补齐邮箱未验证、重置密码和退出所有设备提示。

验收：
- 未验证邮箱不能进入团队邀请、云端 API Key 管理等敏感能力。
- 连续登录失败会被短时间限制。
- 用户可以主动清除其他设备 Session。

## P1R2-2：PostgreSQL 真实存储

要做：
- 为 `PostgresStore` 接入轻量数据库驱动。
- 执行 `server/migrations/001_initial.sql`。
- 增加 JSON Store 到 PostgreSQL 的一次性导入脚本。
- 增加数据库连接健康检查。
- 后端测试增加临时数据库或存储替身策略。

验收：
- `STORE_DRIVER=postgres` 时文档、执笔人、版本、用量、审计都能真实读写。
- 本地 JSON Store 仍然可用。
- 迁移失败时服务明确报错，不会静默降级到错误数据源。

## P1R2-3：团队管理增强

要做：
- 支持邀请撤销和重新发送。
- 支持成员角色调整。
- 支持移除成员。
- 前端团队协作区展示成员角色、邀请状态和基础操作。
- 所有团队管理操作写入审计日志。

验收：
- owner/admin 可以管理成员，member 不能管理成员。
- 被移除成员不能继续访问组织数据。
- 审计日志可以追踪邀请、角色调整和移除动作。

## P1R2-4：部署与可观测

要做：
- 增加 Dockerfile 或最小部署说明。
- 增加 `/api/ready` readiness check，校验存储和关键环境变量。
- 增加结构化请求日志。
- 增加最近错误查询接口，至少覆盖 AI 代理错误和登录失败。
- 增加备份说明或脚本占位。

验收：
- 生产环境缺少关键配置时 readiness 失败。
- 可以定位最近的 AI 供应商错误、接口错误和登录失败。
- 部署文档能指导一个小团队完成内测部署。

## P1R2-5：合规与数据删除

要做：
- 补充隐私政策和用户协议草案。
- 增加用户数据导出说明。
- 增加账号或组织数据删除流程说明。
- 在导入训练样本、云端保存 API Key、邀请成员前补充必要提示。

验收：
- 使用者能理解哪些数据留在本地、哪些数据会进入云端。
- 用户可以找到导出和删除数据的路径。
- 训练样本相关隐私风险有明确提示。

## P1R2-6：灰度试用准备

要做：
- 增加内测用户反馈入口。
- 增加灰度试用清单。
- 增加故障回退说明：云端不可用时如何继续本地使用。
- 更新 README、上手指南、架构文档和 REVIEW。

验收：
- 可以邀请 3 到 5 个真实用户试用。
- 出现后端故障时，用户知道如何回到本地模式继续编辑。
- 每轮试用反馈可以进入 TODO 或 issue。
