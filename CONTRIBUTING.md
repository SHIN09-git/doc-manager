# 贡献指南

欢迎为摹文拟笔工作台提交问题、建议和代码。这个项目更重视“能被真实文档工作长期使用”的稳定性，因此请尽量让每次改动小而清楚。

## 本地开发

```bash
npm install
npm run build
node e2e/serve-static.mjs
```

访问：

```text
http://127.0.0.1:4173/index.html
```

常用检查：

```bash
npm run build
npm run check
npm test
npm run test:e2e
```

## 提交前检查

提交前请至少运行：

```bash
npm run build
npm run check
npm test
```

如果修改了页面结构、拖拽、弹窗、导入导出、执笔人或 PPT 流程，请同时运行：

```bash
npm run test:e2e
```

## 代码约定

- 保持静态前端可运行，不引入必须部署后端的依赖。
- 优先复用现有模块和工具函数。
- 新增交互要考虑键盘可达和小屏幕布局。
- 修改 UI 时复用 `styles.css` 顶部的设计 token，不随意引入新色板。
- 文案统一使用中文，产品概念统一称为“执笔人”。
- 内部兼容历史数据时可以保留 `skill/style` 命名，但不要把它暴露到 UI。
- 修改 `src/` 后必须重新执行 `npm run build`，因为 `index.html` 加载的是 `build/bundle.js`。

## 执笔人相关改动

执笔人是核心能力。相关改动请特别注意：

- 单篇样本只能产生候选规则，不能直接成为强规则。
- 多篇样本聚合时要区分共同规则、差异规则、冲突规则和个案信息。
- 人名、时间、地点、活动名称、临时安排、一次性政策不能写成通用规则。
- 生成结果需要保存版本，失败状态也要能被用户理解和重试。
- `@调用名`、右键段落改写、默认执笔人下拉和导入导出包都要保持兼容。

## 文档导入相关改动

导入能力涉及用户真实资料，请尽量保守：

- 新增格式要有失败提示，不能静默吞掉内容。
- 表格应尽量转为 Markdown 表格或可读文本。
- 编码识别失败时要给用户可理解的建议。
- 删除文档应先进入垃圾箱，除非用户明确选择彻底清除。

## PPT 相关改动

PPT 生成目标是原生可编辑 `.pptx`。

请避免只生成截图式幻灯片。新增版式时应确认：

- 标题和正文不溢出。
- 图表和表格在 16:9 页面内可读。
- 自动页数逻辑不会生成大量空页。
- HTML 预览和 `.pptx` 导出大体一致。

## Issue 建议

报告问题时，请尽量提供：

- 浏览器和系统版本
- 打开方式：直接打开 `index.html` 还是本地服务
- 复现步骤
- 导入文件类型
- 是否配置了 AI 接口
- 控制台错误截图或文字

涉及敏感文档时，请先脱敏，不要上传真实姓名、联系方式、账号、内部安排或密钥。

## Pull Request 建议

PR 描述建议包含：

- 改了什么
- 为什么要改
- 验证了哪些命令
- 是否影响旧数据
- 是否需要更新文档

如果是 UI 改动，最好附一张截图或录屏说明关键状态。
