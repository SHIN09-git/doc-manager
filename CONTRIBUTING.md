# 贡献指南

感谢你愿意参与摹文拟笔工作台。项目目标是做一个本地优先、开箱可用、适合中文正式文本写作与文风沉淀的开源工作台。

## 开发环境

需要：

- Node.js 18+
- npm
- 现代浏览器，推荐 Chrome 或 Edge

安装依赖：

```bash
npm install
```

常用命令：

```bash
npm run build
npm run check
npm test
npm run test:e2e
```

浏览器实际加载 `build/bundle.js`。修改 `src/` 后请运行 `npm run build`，并把打包产物一并纳入提交。

## 项目边界

当前项目是静态前端应用：

- 默认不引入后端服务。
- 默认不内置 API Key。
- 默认以浏览器 IndexedDB 保存工作台数据。
- 真实文件夹关联依赖浏览器 File System Access API。

如果贡献涉及后端代理、云同步、多用户协作或账户系统，请先开 Issue 讨论边界和隐私模型。

## 代码约定

- 用户可见文案使用“执笔人”，内部既有 `skill` 命名可以保留，避免无意义的大规模重命名。
- 优先保持模块边界清晰：AI、文档、文件夹、PPT、执笔人、UI、工具函数分别放在对应目录。
- 新增业务能力时优先走 `eventBus` 或模块注入，不要让业务模块直接依赖大量全局函数。
- AI 调用链路要保留失败重试、进度提示、友好错误和 JSON 容错。
- 文件导入要先走格式验证和友好错误，不要让读取失败静默发生。
- 涉及用户输入渲染到 HTML 时必须转义。

## 执笔人规则

执笔人构建必须坚持：

- 单篇样本只能产生候选规则。
- 多篇共同验证后才可以成为强规则。
- 具体人名、时间、地点、活动名称、临时安排、一次性政策和隐私信息不能提升为通用规则。
- 质量报告要保留规则置信度、冲突提示、个案排除和隐私过滤信息。

## PPTX 规则

PPT 生成功能目前采用：

```text
AI 生成幻灯片 JSON 结构 -> 前端打包为原生 .pptx
```

贡献 PPT 功能时请注意：

- 不要把网页截图当作 PPT 输出。
- 尽量使用原生文本框、表格、形状和备注，保持 PowerPoint / WPS 可编辑。
- 结构 JSON 要做容错归一化，避免 AI 小格式偏差导致下载失败。
- 新增 PPT 布局时要同步更新结构自检、预览归一化和 Playwright 覆盖。
- 新增布局或风格时补充 `test/pptxBuilder.test.js`。

## 测试要求

提交前至少运行：

```bash
npm run check
npm test
```

如果涉及浏览器交互，请额外运行：

```bash
npm run test:e2e
```

首次运行 E2E 前如提示缺少浏览器，请执行 `npx playwright install chromium`。同时建议手动验证：

- 直接打开 `index.html`
- 通过本地服务器打开页面
- 文档导入和拖拽导入
- 执笔人构建和 `@` 调用
- PPTX 草稿生成和下载
- 备份导出

## 文档要求

下面情况需要同步更新文档：

- UI 入口变化，例如接口按钮位置、栏目名称。
- 支持格式变化，例如新增 Office 文件读取。
- AI 链路变化，例如提示词结构、JSON schema、重试策略。
- 存储策略变化，例如 IndexedDB、localStorage fallback、备份格式。
- 开源使用方式变化，例如安装、构建或运行命令。

优先更新：

- `README.md`
- `GETTING_STARTED.md`
- `SECURITY.md`
- `TODO.md`

## 提交建议

- 一个 Pull Request 尽量解决一类问题。
- UI 修改请说明交互变化，最好附截图。
- AI 提示词修改请说明输出结构、隐私过滤或规则置信度的影响。
- 存储结构变化请说明是否需要迁移旧数据。
- 依赖变化请说明用途和浏览器端打包影响。

## Issue 反馈

请尽量提供：

- 运行方式：直接打开 `index.html` 或本地服务器
- 浏览器与系统版本
- 复现步骤
- 预期结果与实际结果
- 控制台错误截图或日志
- 涉及的文件格式和示例文件信息，注意不要上传敏感原文
