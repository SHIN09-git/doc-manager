# 摹文拟笔工作台商业化行动说明

更新时间：2026-05-29

本文用于梳理当前项目脉络，并回答三个问题：

1. 完成这个产品商业化，项目负责人需要做什么。
2. 完成这个产品商业化，接下来的施工计划是什么。
3. 在你提供相关资源的前提下，Codex 可以继续完成什么。

## 一、项目脉络

摹文拟笔工作台目前已经从一个本地文档编辑工具，演进为“文档生成 + 执笔人能力 + PPT 生成 + 云端账户 + 后台管理”的浏览器端工作台。

### 1. 面向普通用户的工作台

普通用户主要在 `index.html` 进入产品，核心能力包括：

- 文档管理：导入、拖拽、阅读、编辑、保存、导出、标签、垃圾箱、备份。
- 文档生成：通过 AI 起草、插入、覆盖当前文档，并支持 `@执笔人` 调用。
- 执笔人工作台：使用、创建、重训、导入、导出、测试和版本管理执笔人。
- PPT 生成：根据文本材料生成原生 `.pptx`，支持页面内预览、编辑和导出。
- 云端页面：展示用户信息、套餐、额度、费用明细、充值申请和数据同步状态。

### 2. 面向管理员的后台

管理员主要在 `admin.html` 进入后台，核心能力包括：

- 用户与组织查看。
- 套餐、额度、用量、成本估算。
- 充值申请人工确认。
- 审计日志、错误记录、用户反馈。
- 负责人、备注、SLA、筛选和运营处理状态。
- 备份、部署和商业化运营相关辅助信息。

### 3. 后端与商业化基础

后端位于 `server/`，目前已经具备商业化雏形：

- 用户注册、登录、会话。
- 管理员权限。
- 套餐与额度模型。
- 用量记录与费用估算。
- 人工确认版充值。
- 审计、反馈、错误记录。
- 本地 JSON 存储和 PostgreSQL 仓储适配方向。
- Docker、部署、CI、测试和生产环境示例配置。

当前状态可以理解为：产品功能和商业化骨架已经成型，适合进入小规模灰度试运营；但距离稳定公开收费，还需要补齐真实生产环境、支付闭环、邮件通知、监控告警、合规和运营流程。

## 二、完成商业化，你需要做什么

以下事项必须由项目负责人准备或最终确认。我可以辅助配置、接入和落地，但不能替你购买、认证或承担法律责任。

### 1. 基础资源

- 域名：购买一个正式域名，例如 `mowen.example.com`。
- 服务器：建议使用云 VPS，而不是长期使用个人电脑。推荐 2 核 4G 起步。
- DNS：建议接入 Cloudflare，方便做解析、HTTPS、基础防护和缓存。
- 备案：如果面向中国大陆用户并使用大陆服务器，需要完成 ICP 备案。
- SSL 证书：可使用 Cloudflare 或 Nginx + Let’s Encrypt。

### 2. 生产环境配置

需要准备并妥善保存这些配置：

- `APP_BASE_URL`：正式网站地址。
- `SESSION_SECRET`：生产会话密钥。
- `ADMIN_EMAIL` / `ADMIN_PASSWORD`：管理员账号。
- `DATABASE_URL`：PostgreSQL 连接地址。
- `OPENAI_API_KEY` 或其他 AI 服务商密钥。
- 邮件服务密钥，例如 Resend、阿里云邮件、腾讯云邮件等。
- 备份存储密钥，例如 S3、R2、OSS、COS。

### 3. 付费与充值资源

你目前设想的是“用户扫码微信或支付宝付款，管理员确认后开通会员或增加额度”。这条路适合早期灰度，所需资源包括：

- 微信收款二维码。
- 支付宝收款二维码。
- 套餐价格表。
- 每个套餐对应的额度规则。
- 管理员确认充值的运营流程。
- 给用户展示的付款说明。
- 退款、误付款、开票和客服处理规则。

后续如果要做自动化支付，还需要：

- 企业主体。
- 微信支付商户号。
- 支付宝开放平台应用。
- 支付回调域名。
- 支付安全证书。
- 订单、回调、对账和退款流程。

### 4. 法务与合规

至少需要你确认这些内容：

- 产品服务条款。
- 隐私政策。
- 用户数据保存周期。
- 用户文档和训练样本是否会被用于模型训练。
- AI 生成内容免责声明。
- 退款规则。
- 发票规则。
- 客服联系方式。
- 若面向学校、公文、组织材料场景，需要特别说明用户应自行审查生成内容。

项目中已有 `PRIVACY_POLICY.md`、`TERMS_OF_SERVICE.md`、`SECURITY.md`，但正式上线前仍建议请专业人士复核。

### 5. 运营决策

商业化不是只有代码，还需要你决定：

- 目标用户是谁：学校文员、行政人员、机构材料岗，还是更广泛的文档工作者。
- 免费额度是多少。
- 套餐分几档。
- 每档包含多少 AI 生成额度、PPT 额度、执笔人数量、存储空间。
- 是否支持团队账号。
- 是否允许用户导出和迁移执笔人。
- 客服响应时间。
- 灰度测试用户名单。

## 三、接下来的商业化施工计划

建议按“可上线、可收款、可运营、可扩展”的顺序继续做，不急着一口气做复杂平台。

### 阶段 A：生产部署封板

目标：让产品可以稳定部署到真实域名。

任务：

- 整理 `.env.production` 模板。
- 运行生产环境配置自检，提前发现占位密钥、缺失数据库、HTTP 地址和未配置收款码。
- 固化 Docker Compose 生产部署。
- 配置 Nginx / Cloudflare 反向代理。
- 检查静态资源、API 地址、跨域和 HTTPS。
- 增加部署后的健康检查。
- 明确本地、测试、生产三套环境的启动方式。

当前已经补齐的工程脚手架：

```bash
copy server\env.production.example server\.env.production
npm run deploy:check -- server/.env.production
npm run deploy:health -- https://api.your-domain.example/api
```

说明：

- `server/src/server.js` 会在 `NODE_ENV=production` 时优先读取 `server/.env.production`。
- `APP_BASE_URL` 已作为 `APP_URL` 的兼容别名，可用于部署平台里更直观地填写正式站点地址。
- `npm run deploy:check` 用于上线前配置检查；它不会替你验证真实账号权限，但会拦截明显的占位值、弱配置和缺失项。
- `npm run deploy:health` 用于部署后检查 `/api/health` 和 `/api/ready`。
- `assets/` 会在 `npm run build:static` 时复制到 `dist/assets/`，可用于放置公开的收款码、Logo 和帮助图片。

验收：

- 用户可以通过正式域名访问工作台。
- 管理员可以通过正式域名访问后台。
- API 健康检查正常。
- 刷新页面不丢失路由和静态资源。

### 阶段 B：生产数据库与备份

目标：从本地 JSON 存储切换到可长期运营的数据库。

任务：

- 选择 PostgreSQL 服务。
- 配置 `DATABASE_URL`。
- 检查现有 repository 是否覆盖用户、套餐、充值、用量、审计、反馈等核心数据。
- 增加迁移脚本或初始化脚本。
- 增加自动备份和恢复演练文档。

当前已经补齐的工程脚手架：

```bash
npm run server:backup
npm run server:backup:verify -- <backup-file>
npm run server:backup:drill -- <backup-file>
npm run server:postgres:import -- <backup-file>
```

说明：

- `server:backup:verify` 会解析明文或加密备份，并检查所有必要数据表是否存在。
- `server:backup:drill` 会把备份恢复到一个临时 JSON Store，再比较恢复前后的表计数；默认不会覆盖任何现有数据。
- `server:postgres:import -- <backup-file>` 可把指定备份导入 PostgreSQL；如果不传备份文件，则沿用本地 `server/.data/db.json`。
- 加密备份需要设置同一个 `BACKUP_ENCRYPTION_KEY` 后再运行校验、演练或导入。

验收：

- 重启服务后用户、额度、充值和审计数据不丢。
- 管理员可以导出关键数据。
- 至少有一套可恢复的备份方案。

### 阶段 C：人工确认版充值完善

目标：先让用户能付款，管理员能确认，额度能到账。

任务：

- [x] 云端页面展示微信 / 支付宝二维码。
- [x] 用户提交充值申请：金额、套餐、支付方式、付款备注、截图或交易号。
- [x] 后台展示充值申请列表，包含订单号、用户、备注、凭证、到账内容和审核备注。
- [x] 管理员确认后自动增加会员时长或额度，拒绝时保留原因。
- [x] 生成充值记录、额度流水和审计日志。
- [x] 给用户显示充值状态和最近费用明细。

验收：

- [x] 用户可以提交充值申请。
- [x] 管理员可以确认、驳回、备注。
- [x] 用户额度或会员状态正确变化。
- [x] 费用明细可追溯。

当前实现说明：

- 用户端：云端页选择套餐和支付方式后，必须填写付款备注或凭证说明；提交后可在“我的充值订单”和“额度明细”里查看状态与到账变化。
- 管理端：独立后台账单页可复制订单详情、填写审核备注、确认或拒绝订单，并查看最近额度流水。
- 数据侧：`manual_payment_orders` 保存订单，`credit_ledger` 保存入账/扣减流水，`audit_logs` 记录订单创建、审核、额度发放和会员开通。

### 阶段 D：邮件与通知

目标：让注册、登录、充值、异常处理有基本通知闭环。

任务：

- 接入邮件服务。
- 注册或登录安全通知。
- 充值申请提交通知。
- 充值确认通知。
- 额度不足提醒。
- 管理员待处理提醒。

验收：

- 关键流程能发邮件。
- 邮件失败不会影响主流程。
- 后台能看到通知失败记录。

### 阶段 E：运营后台增强

目标：让管理员可以日常管理业务，而不是只看数据。

任务：

- 用户详情页。
- 手动调整额度。
- 套餐变更记录。
- 充值处理 SLA。
- 错误和反馈分派。
- 审计日志导出。
- 成本估算和用量趋势。

验收：

- 管理员能处理真实用户问题。
- 每次人工操作都有审计记录。
- 能看出用户是否健康使用产品。

### 阶段 F：灰度上线

目标：找少量真实用户试用，验证收费、体验和成本。

任务：

- 准备灰度测试说明。
- 准备用户使用手册。
- 准备客服话术。
- 准备问题反馈入口。
- 观察 AI 成本、生成成功率、充值转化率。
- 根据反馈修复高频问题。

验收：

- 至少 5-20 位真实用户完成注册、使用、充值或模拟充值。
- 管理员能处理他们的问题。
- 产品能稳定运行一周以上。

## 五、当前优先级建议

如果你想尽快把产品放到真实网站上，建议下一步不要继续大改功能，而是先做这三件事：

1. 确定部署路径：VPS + Docker Compose + Cloudflare + Nginx。
2. 确定商业闭环：人工确认充值 + 套餐额度 + 管理员确认。
3. 确定数据安全：PostgreSQL + 定时备份 + 管理员审计。

这三件事完成后，产品就可以进入小规模灰度试运营。等有真实用户和真实付款意愿后，再接入自动支付、团队空间、高级权限和更复杂的数据分析。

## 七、推荐下一步
