# P2 第二轮商业化施工文档

本文件承接 `P2_ROUND1_COMMERCIALIZATION_BUILD_PLAN.md`。第一轮已经补齐邮件投递、签名支付 webhook、管理汇总 API 和备份脚本；第二轮目标是把“能灰度运营”推进到“能更稳地试收费和团队部署”。

## 施工顺序

```text
1. 云端管理后台二级页面
2. 邮件模板与验证码限流
3. 支付事件适配层
4. 备份恢复演练
5. 组织级数据治理
6. 文档与测试更新
7. 施工后 review
```

## P2R2-1：云端管理后台二级页面

要做：
- 把当前云端面板里的“管理汇总”升级为独立二级页面或模态工作台。
- 分区展示组织信息、成员邀请、用量、审计、反馈、最近错误和账单事件。
- 增加反馈状态入口，为后续“待处理/处理中/已解决/关闭”做铺垫。
- 增加用量和审计筛选 UI，对接已有后端过滤参数。

验收：
- owner/admin 可以从云端面板打开管理后台。
- member 看不到管理后台入口或访问时得到清晰提示。
- 用量和审计至少支持日期、类型、状态筛选。

## P2R2-2：邮件模板与验证码限流

要做：
- 把邮箱验证、密码重置邮件模板独立成可维护函数。
- 增加邮箱验证和密码重置申请的频率限制。
- 邮件投递失败时返回用户友好提示，不泄露内部服务错误。
- 文档说明如何接入真实邮件服务商 webhook。

验收：
- 同一邮箱短时间重复申请会被限制。
- 生产模式仍不会在 API 响应中返回验证码或重置码。
- 邮件模板中不包含额外敏感数据。

## P2R2-3：支付事件适配层

要做：
- 在现有 HMAC webhook 之上增加轻量 payment adapter。
- 统一解析 `checkout.completed`、`subscription.updated`、`subscription.deleted`、`invoice.payment_failed`、`refund.created` 等事件。
- 支付失败和退款先写入系统事件，不立即做激进降级。
- 为后续接 Stripe、Paddle、Lemon Squeezy 或国内支付渠道预留映射点。

验收：
- 支付成功、订阅更新、取消订阅、支付失败、退款都有测试覆盖。
- plan 变更只在可确认组织和套餐时发生。
- 幂等逻辑继续按 `provider + event_id` 生效。

## P2R2-4：备份恢复演练

要做：
- 在 `server:backup` 基础上增加恢复演练脚本或校验脚本。
- 至少校验备份 JSON 可解析、包含必要表、关键数组结构正确。
- 输出恢复演练报告，不直接覆盖线上数据。

验收：
- `npm run server:backup` 生成备份。
- 新增校验命令可以验证备份文件结构。
- 文档说明恢复流程和人工确认点。

## P2R2-5：组织级数据治理

要做：
- 明确组织数据导出的范围。
- 增加组织级导出接口草案，仅 owner/admin 可用。
- 增加组织删除或停用的施工方案，不立即做不可逆删除。
- 更新隐私政策和用户协议草案中的数据删除说明。

验收：
- 个人导出与组织导出的边界清晰。
- 组织级导出不包含 API Key 明文。
- 所有危险操作都有确认或草案状态。

## 非目标

本轮不做：
- 完整商业支付结算页面。
- 企业级 RBAC 权限矩阵。
- PostgreSQL 表级 repository 大重构。
- 自动化恢复线上数据库。

这些进入 P2 第三轮或 P3。

## 完成记录

完成时间：2026-05-23

已完成：

- 反馈状态流转：新增反馈状态更新接口，支持待处理、处理中、已解决和关闭。
- 邮件模板与限流：邮箱验证和密码重置邮件模板集中生成，同一邮箱短时间重复申请会被限制。
- 支付事件适配层：统一解析支付成功、订阅更新、取消订阅、支付失败和退款事件。
- 支付失败与退款策略：仅写入系统事件，不做激进降级。
- 备份恢复演练：新增 `npm run server:backup:verify -- <backup-file>`，校验备份 JSON 和必要表结构。
- 组织级数据治理：新增组织数据导出接口，不返回 API Key 明文；新增组织删除/停用草案接口，不执行不可逆删除。

施工后 review 结论：

- 当前实现可以支撑更完整的灰度运营，但还没有接具体邮件服务商和支付渠道 SDK。
- 组织导出已排除密码哈希和 API Key 密文，只返回 `key_hint`。
- 删除/停用仍为草案状态，避免误删生产数据。
- PostgreSQL Store 仍是快照式读写，下一阶段仍需生产级 repository 改造。
