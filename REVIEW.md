# 代码与架构 Review

日期：2026-05-18

## 结论

当前项目已经从早期单文件原型进入了可维护的模块化阶段：文档、文件夹、AI、执笔人、PPT、存储、UI 与工具函数都已拆到 `src/` 下，测试覆盖了主要基础能力。整体方向是健康的，下一阶段的重点不是推倒重来，而是继续压缩 `app.js` 的编排体积，并把开源协作所需的工程自动化补齐。

## 本轮已处理

- 清理 `src/modules/ppt/guizangPpt.js` 中遗留的旧 PPT 风格函数和旧预览主题常量。PPT 风格现在只从 `src/modules/ppt/pptStyles.js` 注册表读取，避免后续维护者误改旧实现。
- 增强 `src/core/eventBus.js`：订阅监听器必须是函数，取消订阅后自动清理空事件槽。
- 补充 EventBus 单元测试，覆盖取消订阅清理和非法监听器防御。
- 新增 `ARCHITECTURE.md`，说明运行入口、模块边界、数据流、执笔人链路、PPT 链路、AI 调用约定、测试策略和后续风险。

## 主要发现

### 1. PPT 风格曾存在双实现

严重度：中

`src/modules/ppt/guizangPpt.js` 已经从 `pptStyles.js` 导入风格提示词和预览主题，但文件底部仍保留旧的 `buildPptStyleGuide`、`buildMagazineGuide`、`buildSwissGuide` 和旧主题常量。运行时不再使用这些代码，但它们会干扰维护者判断。

状态：已修复。

### 2. EventBus 缺少输入防御和空槽清理

严重度：低

`EventBus.on` 原本允许传入非函数监听器，错误会延迟到 `emit` 时暴露；取消订阅后也会保留空数组。对当前功能影响不大，但长期运行和后续扩展时不够干净。

状态：已修复。

### 3. app.js 仍是主要复杂度集中点

严重度：中

`app.js` 负责 DOM 绑定、状态初始化、跨模块流程、AI 起草、PPT 生成、执笔人构建、布局拖拽等编排工作。模块已经拆出，但控制层还可以继续细分。

建议：后续按功能拆出 controller，例如：

- `generationController`：AI 起草、@执笔人解析、段落重写。
- `pptController`：PPT 面板事件、生成、预览、下载。
- `layoutController`：工作台分栏拖拽、静态选择项填充。
- `apiSettingsController`：接口配置渲染、保存和测试。

### 4. 浏览器端 API Key 是开源版本的安全边界

严重度：中

当前项目定位是本地优先静态应用，API Key 由用户自行配置并保存在本机浏览器。这个设计适合个人本地使用，但不适合多人共享部署。

建议：保留当前简单模式，同时新增一份可选后端代理方案文档，说明生产环境如何让浏览器只访问自己的代理服务。

### 5. Office 文件能力应继续明确边界

严重度：低

`.docx` / `.pptx` 读取目前以文本、表格和备注提取为主，`.docx` / `.pptx` 导出强调可编辑结构，不做完整排版保真。

建议：在 README 和 Issue 模板中持续强调“文本理解和可编辑输出”，避免用户误以为这是完整 Office 格式转换器。

## 后续优先级建议

- 增加 GitHub Actions，至少自动运行 `npm run check` 和 `npm test`。
- 新增 Issue 模板和 PR 模板，降低开源协作沟通成本。

P1：
