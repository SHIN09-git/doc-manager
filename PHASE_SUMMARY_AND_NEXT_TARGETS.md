# 阶段总结与下一阶段施工目标

更新时间：2026-05-24

本文件用于快速说明“摹文拟笔工作台”当前已经完成的建设成果，以及下一阶段应优先推进的施工目标。详细施工步骤继续以 `P2_ROUND4_COMMERCIALIZATION_BUILD_PLAN.md` 为准。

## 一、当前项目状态

项目已经从“本地优先的文档/执笔人工具”推进到“具备小团队灰度试用条件的商业化原型”。

当前可用能力：

- 本地模式完整可用：不登录也能管理文档、编辑正文、AI 起草、构建执笔人、生成 PPT、导入导出和备份。
- 云端模式初步可用：支持账号、组织、团队、文档同步、执笔人同步、云端 AI 代理、用量统计、审计和灰度反馈。
- 商业化后端已成形：具备基础安全、组织隔离、API Key 加密、邮件投递、支付 webhook、备份脚本和管理后台。
- 工程验证已稳定：构建、语法检查、单元测试、后端 API 测试、端到端测试和依赖审计均已通过。

- 依赖审计：0 个漏洞
- 备份成功、备份结构校验、备份失败告警路径均已验证

## 二、已完成工作总结

### 1. 本地工作台能力

- 文档管理：支持导入、拖拽、真实文件夹关联、自定义标签、排序、垃圾箱恢复和彻底清除。
- 文档编辑：支持标题/正文编辑、查找替换、撤销、保存提示、默认 Word 导出和重复标题清理。
- 文件阅读：支持 `.txt`、`.md`、`.csv`、`.docx`、`.pptx`，并对中文编码、Word 表格、PPTX 文本做了兜底处理。
- AI 起草：支持新建文档、插入到当前文档、覆盖当前文档、长耗时进度、取消和友好错误提示。
- PPT 生成：支持原生 `.pptx` 导出、多种默认风格、自定义页数、自动页数和 HTML 放大预览。

### 2. 执笔人核心能力

- UI 从“训练表单优先”调整为“使用执笔人优先”的工作台。
- 执笔人以卡片形式展示，支持调用、编辑说明、重训、测试、详情、导入和导出。
- 支持在提示词中用 `@调用名` 调用执笔人。
- 执笔人生成链路已拆成：单篇分析、多篇聚合、草案生成、人工校准、版本管理、生成测试、反馈优化。
- 对执笔人生成可靠性做了轻量加固：规则证据、混样本警告、反馈事实隔离、JSON 修复、三类测试和运行时优先级。

### 3. 云端商业化底座

- 新增商业化后端 `server/`，支持账号、组织、成员、文档、执笔人、版本、API Key、AI 代理、用量和审计。
- API Key 使用 AES-GCM 加密保存，接口只返回 `key_hint`。
- 云端 AI 代理支持任务类型统计、请求限额、错误记录和审计。
- 生产环境要求强密钥、明确 CORS、生产邮件投递模式，避免默认配置上线。
- 支持 JSON Store 和 PostgreSQL Store，PostgreSQL 已具备迁移和 JSON 导入脚本。

### 4. 团队与账号安全

- 支持注册、登录、邮箱验证、忘记密码、重置密码、退出所有设备。
- 登录失败会被节流。
- 支持组织成员邀请、撤销邀请、重发邀请、接受邀请、角色调整和成员移除。
- 成员移除会清理其云端会话。
- 支持当前用户数据导出和账号删除。

### 5. 运营、支付与备份

- 管理后台二级页面已可查看组织、成员、邀请、用量、审计、反馈、错误和账单事件。
- 管理后台支持用量/审计筛选、反馈状态流转、CSV 导出、错误详情复制和高风险操作确认。
- 邮件投递支持开发日志模式和生产 webhook 模式，并记录投递状态、重试次数和错误摘要。
- 支付 webhook 支持 HMAC 签名、时间戳、防重放、幂等处理、订阅事件、支付失败和退款记录。
- 新增 `PAYMENT_PLAN_PRICE_MAP`，用渠道价格 ID 映射内部套餐，避免伪造 `plan` 字段。
- 备份脚本支持逻辑备份、保留天数、结构校验、失败非零退出和失败 webhook 告警。

### 6. 文档与开源准备

- 已补齐 README、架构、部署、数据库、隐私政策草案、用户协议草案、灰度清单、商业化计划和多轮施工记录。
- 已新增 PostgreSQL Store 表级 repository 改造方案。
- 已新增下一阶段施工文件 `P2_ROUND4_COMMERCIALIZATION_BUILD_PLAN.md`。

## 三、当前剩余问题

虽然项目已经可以灰度试用，但还不建议直接大规模收费上线。主要缺口如下：

- 支付链路还没有正式 checkout、账单页、支付渠道 SDK 和发票体系。
- 邮件链路还没有服务商状态回调、退信处理和运营筛选闭环。
- 管理后台仍是内嵌弹窗，不适合作为正式运营后台长期使用。
- PostgreSQL Store 仍保留快照式兼容层，高并发多租户前需要表级 repository 改造。
- 备份还缺加密上传、对象存储和正式恢复演练。
- 合规文档仍是草案，正式商业化前需要按真实主体、服务边界和数据流重写。

## 四、下一阶段总目标

下一阶段建议进入 P2 第四轮：从“可灰度试用”推进到“可收费灰度”。

核心目标：

1. 让用户能看懂套餐和账单状态，并有安全的升级入口。
2. 让邮件投递、支付事件、备份和管理后台形成更完整的运营闭环。
3. 开始把 PostgreSQL Store 从灰度可用推进到生产可扩展。
4. 为后续正式商业化、独立管理后台和对象存储备份打基础。

## 五、下一阶段施工目标

### P2R4-1：支付 checkout 与账单页入口

目标：

- 新增云端账单页入口，展示当前套餐、额度、用量、升级按钮和账单事件。
- 后端新增创建 checkout session 的适配接口。
- 前端升级按钮只调用后端，不直接拼接第三方支付链接。

验收标准：

- 免费用户能看到清晰升级入口。
- 支付配置缺失时返回友好提示，不暴露内部错误。
- checkout 接口能为真实支付渠道预留 provider adapter。

### P2R4-2：邮件投递回调与运营筛选

目标：

- 增加邮件服务商回调接口草案。
- 支持将邮件投递状态更新为 delivered、bounced、failed 等。
- 管理后台增加邮件投递列表和筛选。

验收标准：

- 管理员能按模板、状态、邮箱查看邮件投递记录。
- 邮件失败可复制详情。
- 回调签名或 token 校验有最小安全保护。

### P2R4-3：管理后台独立页面雏形

目标：

- 在保留现有云端面板入口的同时，增加独立管理页面或 hash 路由。
- 将组织概览、成员、用量、审计、反馈、错误、邮件、账单拆成更清晰的二级视图。
- 保持 owner/admin 权限判断。

验收标准：

- 管理后台不再受右侧面板空间限制。
- 原管理入口仍可进入或跳转。
- 普通成员仍无法访问管理后台。

### P2R4-4：PostgreSQL repository 试点

目标：

- 按 `POSTGRES_REPOSITORY_REFACTOR_PLAN.md` 实现迁移版本表。
- 先为 `ai_usage` 或 `audit_logs` 做一个只读 repository 试点。
- 保持 JSON Store 不受影响。

验收标准：

- PostgreSQL Store 启动时能记录迁移版本。
- 试点 repository 有测试覆盖。
- 现有后端 API 测试全部继续通过。

### P2R4-5：备份加密与对象存储方案

目标：

- 出具对象存储和加密备份配置方案。
- 备份脚本预留加密或上传接口。
- 未配置对象存储时仍保持本地备份行为。

验收标准：

- 文档明确备份文件不应明文长期落盘。
- 备份恢复前的密钥、权限和只读校验步骤清楚。
- 脚本向后兼容，不破坏当前 `server:backup`。

## 六、建议施工顺序

建议按这个顺序推进：

1. 先做支付 checkout 与账单页入口。
2. 再做邮件投递回调和邮件运营筛选。
3. 然后拆管理后台独立页面雏形。
4. 接着做 PostgreSQL repository 只读试点。
5. 最后补备份加密和对象存储方案。
6. 完成后统一 review、跑完整测试，并更新下一阶段施工文档。

这样安排的原因是：支付和邮件最接近“可收费灰度”的真实用户路径；管理后台承接运营观察；数据库和备份是生产化底层能力，适合在业务闭环明确后逐步加固。

## 七、下一阶段非目标

下一阶段暂不做：

- 完整发票系统。
- 企业级工单系统。
- PostgreSQL 全量 repository 改造。
- 多租户高并发压测平台。
- 执笔人市场或社区分发平台。
- 大规模公开注册和正式 SLA 承诺。
