# 网站上线未完成工作与 TODO

更新时间：2026-06-09

本文档面向一个明确目标：把“摹文拟笔工作台”从本地可用项目推进到真实网站，让用户可以通过域名访问、注册登录、使用 AI、提交人工确认版充值，并由管理员在后台完成运营管理。

## 当前结论

项目已经具备灰度试运行的产品底座，但还没有完成公开收费网站上线所需的真实资源接入、生产联调、运营规则确认和上线演练。

可以概括为：

- 产品功能已经较完整：文档管理、AI 起草、执笔人构建、PPT 生成、云端账号、管理员后台、人工充值链路均已具备基础能力。
- 工程底座已经较完整：前端静态构建、Node API、PostgreSQL Store、Docker、备份、邮件适配、测试和 GitHub Actions 均已具备基础方案。
- 还缺真实上线资源：域名、服务器、生产数据库、AI Key、邮件服务、收款码、套餐规则、客服方式、合规文本确认、备份恢复演练和真实环境验收。
- 当前最合适的上线方式是小范围灰度：先用人工确认版充值跑通 3 到 20 个真实用户，不急着接自动支付。

## 已完成的基础能力

### 用户侧

- 文档管理：导入、拖拽、真实文件夹关联、标签、排序、垃圾箱、Word 默认导出。
- 文本编辑：查找替换、撤销、右键段落改写、格式整理、导出前避免重复标题。
- AI 起草：新建文档、插入、覆盖当前文档、取消生成、进度提示、失败重试和友好错误提示。
- 执笔人：卡片化使用、`@调用名` 调用、训练样本导入、重训、版本、测试、导入导出、右键段落改写。
- PPT：原生 `.pptx` 生成、自动页数、自定义风格、预览、素材拖入。
- 云端页面：账号信息、同步、套餐、额度、人工充值、费用明细、反馈入口。

### 管理侧

- 独立后台 `admin.html`。
- 用户、组织、成员、角色、邀请管理。
- 组织 API Key 加密保存。
- AI 用量、成本估算、审计日志、错误记录、反馈处理。
- 人工充值订单：用户提交，管理员确认或拒绝，确认后额度或会员到账。
- 管理员、普通成员、运营只读角色边界。

### 工程侧

- 静态前端构建：`npm run build:static`。
- Node API 后端：`server/src/server.js`。
- JSON Store 本地开发路径。
- PostgreSQL Store 与迁移脚本。
- Dockerfile、Docker Compose、Render 草案。
- 生产环境变量校验。
- 邮件服务适配：通用 webhook 与 Resend。
- 备份、加密备份、备份校验和对象存储预留。
- GitHub Actions 与测试体系。

### P1：灰度上线后尽快完成

目标：降低运营成本，让第一批真实用户更顺畅地完成注册、充值、使用和反馈。

- 新用户引导：注册、邮箱验证、云端同步、充值、额度不足提示形成连续路径。
- 站内通知：充值提交、审核通过、审核拒绝、额度不足、邮箱状态变化都能主动提示。
- 用户帮助：快速上手、充值说明、执笔人训练说明、PPT 生成说明、常见问题。
- 管理员帮助：充值审核、用户处理、反馈处理、错误处理、备份恢复说明。
- 监控告警：API 5xx、AI 失败、邮件失败、备份失败、数据库容量、额度异常。
- 运营日报：每日注册、活跃、用量、充值、成本、错误、反馈摘要。
- 默认云端地址：部署后前端默认使用正式 `/api`，减少用户手动配置。

### P2：公开收费版本需要完成

目标：从人工灰度运营升级为更可规模化的商业网站。

- 自动支付：至少接入一种微信、支付宝、Stripe、Paddle 或 Creem 等支付渠道。
- 支付对账：订单号、渠道回调、退款、重复回调幂等、内部额度流水一致性。
- 发票与财务：发票规则、账单导出、收入记录、税务处理流程。
- 客服工单：用户问题编号、处理状态、负责人、SLA 和消息通知。
- 团队套餐：组织共享额度、成员权限、团队执笔人共享、团队文档空间。
- 数据治理：数据保留周期、组织删除、账号删除、样本脱敏、敏感材料提醒。
- 安全运营：生产密钥轮换、漏洞反馈渠道、安全响应流程、访问日志留存策略。
