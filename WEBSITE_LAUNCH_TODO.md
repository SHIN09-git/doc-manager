# 网站上线未完成工作与 TODO

更新日期：2026-06-09

本文档用于统筹“摹文拟笔工作台”从本地可用产品走向真实网站灰度上线的剩余工作。当前目标不是一步到位做成完整 SaaS，而是先上线一个可控版本：用户能通过域名访问、注册登录、使用 AI、提交人工充值、查看额度和费用明细；管理员能确认充值、处理反馈、排查错误、查看用量成本并维护数据安全。

## 1. 当前结论

项目已经具备灰度试运营底座，但还不建议直接大规模公开收费。

已经完成：

- 工作台核心功能：文档管理、文本编辑、AI 起草、执笔人构建与调用、PPT 生成、本地备份、云端账户、套餐额度、费用明细、人工充值申请。
- 管理后台：独立 `admin.html`，支持组织、成员、接口、用量、成本估算、审计、反馈、错误跟进、邮件记录和人工充值审核。
- 后端基础：注册登录、会话、邮箱验证、密码重置、组织、额度、套餐、人工充值、AI 代理、邮件适配、审计、备份、PostgreSQL Store、Docker 和部署检查。
- 工程质量：已有构建检查、单元测试、端到端测试、GitHub Actions、生产环境变量校验、敏感字段脱敏、备份校验和恢复演练脚本。

仍未完成：

- 真实生产入口没有落地：正式域名、DNS、HTTPS、主站地址和 API 地址还未接入。
- 真实生产环境没有跑通：VPS 或托管平台、生产 PostgreSQL、Nginx/反向代理、生产环境变量还未在真实机器上验收。
- 第三方服务没有真实联调：AI Key、邮件服务、收款码 HTTPS 图片地址、邮件投递回调、备份告警都需要用真实账号验证。
- 商业规则没有最终确认：套餐价格、额度扣减、充值到账、退款规则、客服入口、运营主体、隐私政策、用户协议、AI 免责声明需要你确认。
- 运营闭环没有真实演练：灰度名单、管理员分工、充值审核流程、每日巡检、备份恢复演练还没有在真实环境执行。

推荐第一版上线形态：

- 部署：`VPS + Docker Compose + Nginx + Cloudflare + PostgreSQL`。
- 充值：先使用微信/支付宝收款码，管理员后台人工确认到账。
- 灰度：邀请 3 到 20 个真实用户试用 1 到 2 周。
- 暂缓：自动支付、发票系统、复杂客服工单、公开营销官网、执笔人市场、复杂团队协作。

## 3. 你需要准备的资源

### 上线入口

- [ ] 正式域名。
- [ ] DNS 管理权限，推荐接入 Cloudflare。
- [ ] 是否面向中国大陆用户；如果使用大陆服务器，需要确认 ICP 备案路径。

### 服务器与数据库

- [ ] 一台 VPS 或云服务器，第一版建议 2 核 4G 起步。
- [ ] 服务器登录方式，建议 SSH 密钥。
- [ ] PostgreSQL 方案：VPS Docker 内置，或云数据库。
- [ ] 数据库备份保存位置：服务器本地、对象存储，或两者都要。

### 第三方服务

- [ ] AI 服务商 API Key、模型名称、预算上限、失败时是否允许重试。
- [ ] 邮件服务账号，推荐 Resend；需要完成发信域名验证。
- [ ] 微信和支付宝收款码图片，或可公开访问的 HTTPS 图片地址。
- [ ] 如果后续要自动支付，准备微信支付、支付宝、Stripe 等商户资料。

### 商业规则

- [ ] 免费版每日额度。
- [ ] Pro 套餐价格、额度、有效期。
- [ ] Team 套餐价格、额度、有效期。
- [ ] 单独额度包价格和额度。
- [ ] 人工充值最低金额。
- [ ] 额度消耗规则：一次 AI 调用扣多少，PPT 是否单独扣费。
- [ ] 退款规则：什么情况下退，退到哪里，额度已消耗如何处理。

### 运营与合规

- [ ] 运营主体名称。
- [ ] 客服邮箱、微信、QQ 群或飞书群。
- [ ] 至少一名管理员负责充值审核。
- [ ] 至少一名管理员负责错误和反馈跟进。
- [ ] 隐私政策、用户协议、AI 生成内容免责声明的最终确认。
- [ ] 第一批灰度用户名单和典型使用场景。
